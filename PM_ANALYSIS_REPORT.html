아키텍처 품질 평가

Clean Architecture 기반, 프로덕션 수준의 코드 구조

구조

Clean Architecture 엄격 적용

Domain / Data / Presentation 3계층 분리. 도메인 레이어에 프레임워크 의존성 없음. Repository 패턴으로 데이터 소스 추상화. 향후 Firebase, 새 AI 서비스 교체 시 도메인 레이어 변경 불필요.

패턴

일관된 디자인 패턴

20개 UseCase 전부 동일 패턴 (suspend operator fun invoke + Result<T>). ViewModel은 StateFlow 기반 반응형 상태 관리. Sealed class로 타입 안전한 UI 상태. Android Hilt / iOS DIContainer로 의존성 주입.

플랫폼

네이티브 듀얼 플랫폼

Android: Kotlin + Jetpack Compose + Room + Glance Widget.
iOS: Swift + SwiftUI + Core Data + WidgetKit.
크로스 플랫폼 프레임워크 없이 각 OS 최적 API 활용. 위젯, 음성 인식, 알림 모두 네이티브.

확장성

오프라인 퍼스트 설계

모든 Entity에 SyncStatus 필드 내장. 로컬 우선 저장 후 클라우드 동기화 설계. AI 제외 옵션 (excludeFromAI). 이 설계를 나중에 추가하는 것은 매우 어려운데, 처음부터 반영된 것은 큰 장점.

아키텍처 평가: 5/5 솔로 개발자 프로젝트치고는 극히 이례적인 수준의 설계 품질. Repository 패턴, UseCase 분리, 반응형 상태 관리 모두 교과서적으로 구현되어 있으며, 향후 팀 확장이나 기능 추가 시 안정적인 기반이 될 것.

SWOT 분석

내부 역량과 외부 환경을 교차 분석한 전략적 진단

S — 강점 (내부)

  • S1. 뛰어난 아키텍처 품질Clean Architecture + MVVM, 150+ 파일에 일관된 패턴. 팀 확장 시 온보딩이 매우 수월.
  • S2. 네이티브 듀얼 플랫폼Kotlin + Swift 각각 최적화. 위젯, STT, 알림 모두 플랫폼 네이티브 API 활용.
  • S3. 혁신적 인터랙션 패러다임"음성으로 입력하고, 대화로 찾는다" — 경쟁 앱들의 수동 입력 + 키워드 검색과 본질적으로 다른 접근.
  • S4. 프라이버시 중심 설계로컬/클라우드 선택, 개별 기록 잠금, AI 제외 토글. 개인 데이터 AI 처리에 대한 우려 선제 대응.
  • S5. 체계적인 문서화PRD v3, 아키텍처 문서, 12개 Stitch 디자인 목업, 구현 현황 문서. 투자자/팀원 온보딩 자료로 즉시 활용 가능.

W — 약점 (내부)

  • W1. AI 서비스 미연동 (시뮬레이션)핵심 가치 제안인 AI 추출/대화형 검색이 더미 응답. 실제 API 연동 전까지 "데모"에 불과.
  • W2. 클라우드 인프라 부재Firebase 프로젝트 미생성. 인증, 동기화, 백업 모두 불가. 현대 앱의 기본 기대치 미충족.
  • W3. 테스트 커버리지 0%10,700줄 코드에 테스트 없음. 리팩토링이나 기능 추가 시 회귀 버그 위험.
  • W4. 솔로 개발자 (Bus Factor = 1)모든 지식과 개발 속도가 한 사람에 집중. 코드 리뷰 프로세스 없음.
  • W5. 실사용자 검증 없음12개 디자인 목업은 있지만 실제 사용자 피드백 0건. 음성 입력 UX가 실제로 작동하는지 미확인.

O — 기회 (외부)

  • O1. "AI 기억 비서" 블루오션Day One은 저널링, Monica는 CRM. "말하면 기록, 물어보면 대답" 카테고리에 지배적 플레이어 없음.
  • O2. AI 비용 급감 + 성능 급상승LLM 비용 연간 10배 하락. 온디바이스 AI (Apple Intelligence, Gemini Nano)로 API 비용 제거 가능성.
  • O3. $5.1B 시장, 9.5-11.5% CAGR개인 생산성 + 저널링 + 관계 관리 시장 수렴. AI가 채택을 가속화.
  • O4. 웨어러블 확장Apple Watch에서 음성 캡처 = 궁극의 저마찰 입력. "손목을 탭하고 10초 말하면 끝."
  • O5. 캘린더/메시징 연동일정 후 자동 프롬프트: "박지영님과 점심 마쳤네요, 기억할 것 있나요?"

T — 위협 (외부)

  • T1. Apple/Google 네이티브 구현Apple Intelligence + 일기장 앱 진화, Google Gemini + Keep 통합. 12-18개월 내 가능.
  • T2. 기존 앱들의 AI 기능 추가Day One(1,000만+ 사용자)이 AI 추출 + 대화 검색 추가 시, 배포망으로 선점.
  • T3. AI API 비용이 마진 잠식GPT-4 호출당 $0.01-0.03. 활성 사용자 월 $2-3 AI 비용 시 $4.99 프리미엄으로 수익 불가능.
  • T4. 개인정보 AI 처리 반발개인 기억을 클라우드 AI에 전송하는 것에 대한 사용자 저항. 데이터 유출 사고 시 대량 이탈.
  • T5. 높은 사용자 획득 비용생산성 앱 CPI $3-8. 프리미엄 전환율 2-5%. 바이럴 메커니즘 없이는 CAC > LTV.

세그먼트 1: 바쁜 직장인 (Primary)

1. WHO

타겟 사용자

30-50대 직장인. 매일 수십 개의 개인적/업무적 관계를 오가며, 세부 사항을 기억하고 싶지만 정신적 여유가 없는 사람들.

2. WHY

해결하려는 과업 (JTBD)

"관계에서 중요한 디테일을 절대 잊지 않기." 동료 자녀 이름, 친구 수술 날짜, 지난 화요일 저녁에 약속한 것 — 바쁜 일상에서도 세심한 관계를 유지하고 싶다.

3. WHAT BEFORE

현재의 고통

메모 앱: 적어놓지만 다시 못 찾음. 캘린더: "언제"는 있지만 "무슨 대화"는 없음. CRM (Monica, Dex): 수동 입력 + 차가운 느낌, 2주 내 포기. 저널링 앱: 글쓰기용이지 검색용이 아님.

4. HOW

솔루션

원탭 음성 캡처: 위젯으로 즉시 녹음. AI 자동 추출: 인물, 장소, 날짜, 금액 자동 태그. 대화형 검색: "철수 결혼 언제야?"로 바로 답변. 스마트 리마인더: 맥락 파악 후 적시 알림.

5. WHAT AFTER

개선된 미래

모든 대화와 약속이 몇 초 만에 기록되고 자동 정리됨. 일상 언어로 질문하면 답을 얻음. 모든 모임, 생일, 후속 조치에 준비된 상태로 나타남. "항상 기억해주는 사람"이라는 평가.

6. ALTERNATIVES

대안 대비 차별점

vs 메모앱: 자동 정리 (수동 X). vs Day One: 검색 우선 (글쓰기 우선 X). vs Monica/Dex: 음성 입력 + AI (수동 폼 X). vs 기억력: 영구적, 검색 가능, 맥락 인식.

핵심 개선 제안 7가지

SWOT 교차 분석 + 가치 제안 검증 기반 전략적 개선안

P0 — 긴급

1. AI 핵심 루프를 실제 API로 연동하고 검증하라

제품의 전체 가치 제안이 AI 추출 정확도에 달려 있습니다. 현재 시뮬레이션 모드로는 핵심 가설("음성→AI 추출→대화 검색"이 사용자를 만족시키는가?)을 검증할 수 없습니다. AI 추출 정확도가 60%면 사용자는 신뢰를 잃고 2주 내 이탈합니다.
실행 항목
  • OpenAI/Claude API 키 발급 및 AIAnalysisService에 실제 연동
  • 한국어 + 영어 음성 100건으로 추출 정확도 테스트
  • 인물/날짜 추출 정확도 85% 이상, 금액/카테고리 75% 이상 목표
  • 실패 시 프롬프트 엔지니어링 반복 최적화
목표 지표: AI 추출 정확도 > 85%
P0 — 긴급

2. 50명 클로즈드 베타로 핵심 가설을 검증하라

14주 개발 후 출시하고 나서 "사용자가 음성으로 기록하지 않는다"를 발견하면 치명적입니다. 음성 입력 UX는 검증되지 않은 인터랙션 패턴입니다. 리텐션 클리프(30일 내 90% 이탈)는 저널링 앱의 고질적 문제이며, DailyMemory도 예외가 아닐 수 있습니다.
실행 항목
  • Android 단독 클로즈드 베타 (로컬 전용, Firebase 없이도 가능)
  • 30-50대 직장인 50명 모집 (지인, 온라인 커뮤니티)
  • 핵심 플로우 계측: 녹음 완료율, AI 추출 수정 비율, 검색 성공률
  • 2주 운영 후 10명 심층 인터뷰
목표 지표: 7일 리텐션 > 30%, 주간 녹음 > 3회
P1 — 높음

3. 기능 해자가 아닌 속도 해자를 구축하라

Day One이 AI 기능을 추가하기까지 6-12개월. Apple Intelligence가 "AI 기억" 기능을 내장하기까지 12-18개월. 이 창이 닫히기 전에 "AI 기억 비서" 카테고리를 선점해야 합니다. 완벽한 앱을 늦게 출시하는 것보다 80% 완성된 앱을 빨리 출시하는 것이 전략적으로 옳습니다.
실행 항목
  • MVP 범위 재정의: 음성 캡처 + AI 추출 + 대화 검색 + 리마인더 (4개 핵심 기능만)
  • Phase 7 (전체 테스트)을 스모크 테스트로 축소 — 속도 우선
  • iOS 우선 출시 (높은 ARPU + 앱스토어 에디토리얼 가능성)
  • 앱스토어 "오늘의 앱" 추천 신청 준비
목표 지표: 8주 내 퍼블릭 런칭
P1 — 높음

4. 유닛 이코노믹스를 출시 전에 검증하라

프리미엄 $4.99/월에서 AI API 비용이 사용자당 $2-3/월이면 마진이 거의 없습니다. 사용자가 하루 2건 기록 + 0.5건 검색 시 월간 AI 비용을 정확히 산출해야 합니다. 스케일링 전에 수익성이 확인되지 않으면, 사용자가 늘수록 적자가 커집니다.
실행 항목
  • 베타 기간 중 사용자당 AI API 호출 수 및 비용 계측
  • 프롬프트 최적화로 토큰 사용량 최소화 (짧은 시스템 프롬프트, 캐싱)
  • 온디바이스 AI (Apple Intelligence, Gemini Nano) 가능성 조사
  • 프리미엄 가격 $4.99 → $6.99 테스트 (지불 의향 조사)
목표 지표: 활성 사용자당 월 AI 비용 < $2.00
P1 — 높음

5. "프라이버시 퍼스트 AI" 내러티브를 브랜드 핵심으로 삼아라

개인 기억(건강, 재정, 관계)을 AI에 보내는 것에 대한 거부감은 실재합니다. 경쟁자들이 클라우드 전용 AI를 밀 때, DailyMemory의 로컬 퍼스트 설계는 강력한 차별점이 됩니다. 이 아키텍처적 결정을 마케팅 무기로 전환하세요.
실행 항목
  • 온보딩에 "당신의 기억은 기기 안에 안전합니다" 메시지 강조
  • 앱스토어 설명에 프라이버시 강점을 첫 번째로 배치
  • 온디바이스 AI 통합 시 "기억이 기기를 떠나지 않습니다" 캠페인 준비
  • Apple App Privacy Labels에서 최소 데이터 수집 강조
목표 지표: 앱스토어 프라이버시 관련 리뷰 평점 > 4.5
P2 — 중간

6. 캘린더 연동으로 "제로 입력" 기억을 구현하라

음성 입력도 여전히 사용자의 의지가 필요합니다. 캘린더 일정 종료 후 자동 프롬프트("박지영님과 점심 끝났네요, 기억할 것 있나요?")를 보내면 기록 마찰이 한 단계 더 줄어듭니다. 이미 아이디어 백로그에 있으나, 리텐션에 직접적인 영향을 주므로 우선순위를 높여야 합니다.
실행 항목
  • Google Calendar / Apple Calendar API 읽기 전용 연동
  • 일정 종료 후 푸시 알림 → 원탭으로 기록 화면 진입
  • 일정 참석자를 Person으로 자동 연결
  • 일정 컨텍스트를 AI 분석에 사전 제공 (정확도 향상)
목표 지표: 일정 후 기록 전환율 > 20%
P2 — 중간

7. 바이럴 루프를 설계하라 — "기억 공유" 기능

생산성 앱 카테고리에서 유기적 성장은 어렵습니다 (CPI $3-8). 바이럴 메커니즘 없이는 유료 광고에 의존하게 되고, CAC가 LTV를 초과할 위험이 있습니다. "함께한 기억"을 공유하는 소셜 기능은 자연스러운 바이럴 루프를 만들 수 있습니다.
실행 항목
  • "이 기억 공유하기" — 특정 메모리를 카드 이미지로 변환해 공유
  • "함께한 기억" — 상대방에게 공동 기록 초대 (커플/가족 모드)
  • "연간 관계 리포트" — 인스타그램 스토리용 연간 요약 카드 자동 생성
  • 추천 프로그램: 친구 초대 시 프리미엄 1개월 무료
목표 지표: K-Factor > 0.3 (초대당 0.3명 신규 유입)
7

개선된 실행 로드맵

기존 Phase 6-7 대신, 검증 + 속도 중심으로 재구성

로드맵 재구성 이유 기존 로드맵은 "기능 완성 → 테스트 → 출시" 순서입니다. 하지만 핵심 가설(AI 추출이 실제로 작동하는가? 사용자가 음성으로 기록하는가?)이 검증되지 않은 상태에서 Phase 6(클라우드)과 Phase 7(테스트)을 진행하는 것은 리스크가 큽니다. "검증 → MVP 출시 → 반복 개선" 순서로 재배치를 권장합니다.
1-2주차 — AI 연동 & 기본 인프라

AI 핵심 루프 실제 연동

• OpenAI/Claude API 실제 연동 (AIAnalysisService)
• Firebase 프로젝트 생성 + 이메일/Google 인증 구현
• AI 추출 정확도 100건 테스트 및 프롬프트 최적화
• Firebase Analytics + Crashlytics 기본 설정

3-4주차 — 클로즈드 베타

50명 베타 테스트 및 핵심 지표 수집

• Android TestFlight/내부 테스트 트랙 배포
• 핵심 플로우 이벤트 트래킹 (녹음 시작/완료, AI 수정 비율, 검색 빈도)
• 기본 Firestore 동기화 (메모리 백업 수준)
• 2주 운영 + 10명 사용자 인터뷰

5-6주차 — 베타 피드백 반영

핵심 이슈 수정 + MVP 다듬기

• 베타 피드백 기반 UX 개선 (가장 많은 불만 top 3 해결)
• AI 추출 프롬프트 2차 최적화
• 핵심 경로 스모크 테스트 작성 (UseCase 10개 + ViewModel 5개)
• 유닛 이코노믹스 분석 (사용자당 AI 비용 산출)

7-8주차 — 퍼블릭 런칭

앱스토어 출시 + 초기 성장

• iOS App Store + Google Play 동시 출시
• 앱스토어 최적화 (ASO): 스크린샷, 설명, 키워드
• 앱스토어 "오늘의 앱" 에디토리얼 추천 신청
• ProductHunt 런칭 + 테크 블로그 홍보

9-12주차 — 성장 + 확장

리텐션 최적화 + 차별화 기능 추가

• 캘린더 연동 (일정 후 자동 프롬프트)
• 시맨틱 검색 (벡터 임베딩 기반)
• 바이럴 기능 (기억 공유 카드, 연간 리포트)
• 프리미엄 구독 인프라 구축 + A/B 가격 테스트

8

핵심 성과 지표 (KPI)

각 단계별 추적해야 할 지표와 목표값

지표 현재 4주 목표 (베타) 12주 목표 (출시 후)
AI 추출 정확도 미측정 (시뮬레이션) > 80% > 90%
7일 리텐션 N/A > 30% > 40%
주간 녹음 횟수/사용자 N/A > 3회 > 5회
AI 검색 사용률 N/A 측정 시작 DAU의 > 20%
활성 사용자당 월 AI 비용 미측정 측정 시작 < $2.00
앱스토어 평점 미출시 TestFlight > 4.5
테스트 커버리지 0% 20% (스모크) 60%
프리미엄 전환율 N/A N/A > 5%
노스스타 메트릭 (North Star Metric) 제안 "주간 활성 기록자 수 (Weekly Active Recorders)"

이유: DailyMemory의 핵심 가치는 "기록 → AI 정리 → 대화형 검색" 루프입니다. 이 루프의 시작점인 "기록"이 일어나지 않으면 나머지 모든 기능이 무의미합니다. 주간 단위가 적절한 이유는, 매일 기록하는 것은 높은 기대치이지만 주 3회 이상 기록하는 사용자는 진정한 활성 사용자이기 때문입니다.

보조 지표: AI 추출 수정률 (낮을수록 좋음), 검색 성공률, 리마인더 완료율
